0. Introducció: Dos Rols, Un Objectiu

En aquesta sessió aprendràs a veure el codi des de dues perspectives diferents però complementàries:

DEVELOPER

Perspectiva del Desenvolupador

Què faig? Escric el codi que fa coses (funcions, classes, mètodes)

Com penso? "Com puc fer que això funcioni correctament i gestioni errors?"

Què creo?

  • Classes i mètodes
  • Gestió d'excepcions (throw)
  • Validacions de dades
  • Documentació (JavaDoc)
TESTER

Perspectiva del Tester

Què faig? Escric tests que verifiquen que el codi funciona

Com penso? "Com puc trencar això? Què pot anar malament?"

Què creo?

  • Classes de test (*Test.java)
  • Verificacions (assertEquals, assertThrows...)
  • Casos de prova
  • Tests d'excepcions

Important

En un projecte real, sovint la mateixa persona fa TOTS DOS rols:

  1. Escrius el codi (rol developer)
  2. Escrius els tests (rol tester)
  3. Els tests fallen i tornes a ser developer i arregles el codi
  4. Repeteixes fins que tot funciona

1. Excepcions: La Base que Has d'Entendre

Què és una excepció?

Una excepció és com un semàfor vermell al codi: s'atura tot quan detecta un problema.

Analogia del món real

Imagina que estàs conduint:

  • Tot correcte: Continua conduint (el codi s'executa normalment)
  • Problema detectat: Semàfor vermell (excepció llançada)
  • Gestió: T'aturs i decideixes què fer (try-catch o deixar que el programa s'aturi)

Tipus d'excepcions en Java

Tipus Quan s'usa Has de gestionar-la?
RuntimeException
(i les seves filles)		 Errors de programació que podries haver evitat: null, divisió per 0, índex incorrecte... NO és obligatori (però recomanable)
Exception
(checked exceptions)		 Problemes previsibles però inevitables: fitxer no existeix, xarxa caiguda... SÍ, obligatori (has de posar try-catch o throws)
Error Problemes greus del sistema: memòria exhaurida, stack overflow... NO té sentit (el programa morirà)

Com funcionen les excepcions?

Quan llances una excepció amb throw new TipusExcepcio():

  1. El mètode actual s'atura immediatament

    No continua executant codi després del throw

  2. L'excepció "puja" per la pila de crides

    Va buscant un try-catch que la capturi

  3. Si no troba cap try-catch

    El programa sencer es para i mostra l'error

  4. Si troba un try-catch

    Executa el bloc catch i continua

2. Triar la Excepció Correcta: La Guia Definitiva

Aquesta és la pregunta que tots els developers novells es fan: "Quina excepció he de llançar?"

Les RuntimeExceptions més comunes

Excepció Quan usar-la Exemple
IllegalArgumentException El valor del paràmetre és invàlid (fora de rang, format incorrecte...) edat = -5, email sense @
NullPointerException Tries usar algo que és null String s = null; s.length()
IllegalStateException L'objecte està en un estat incorrecte per fer aquesta operació Tries llegir d'un fitxer tancat
ArithmeticException Error en una operació matemàtica Divisió per zero
IndexOutOfBoundsException Índex fora dels límits array[10] quan array té 5 elements
UnsupportedOperationException Operació no suportada Tries modificar una llista immutable

Regla d'or per triar

  1. És un paràmetre invàlid?

    Usa IllegalArgumentException

  2. És null i no hauria de ser-ho?

    Usa NullPointerException (o IllegalArgumentException si és un paràmetre)

  3. És un problema matemàtic?

    Usa ArithmeticException

  4. És un problema d'estat de l'objecte?

    Usa IllegalStateException

  5. Res d'això?

    Usa RuntimeException (genèrica) o crea una excepció personalitzada

Per què importa triar bé?

Triar la excepció correcta ajuda a:

  • Claredat: Qui llegeix el codi entén immediatament quin és el problema
  • Debugging: És més fàcil trobar errors quan les excepcions són específiques
  • Testing: Els tests poden verificar excepcions concretes amb assertThrows
  • Documentació: El JavaDoc és més precís quan especifiques quin tipus d'excepció

3. Exemple Complet: Developer vs Tester

Vegem el mateix codi des de les dues perspectives amb TOTS els detalls.

Objectiu: Validador d'Edat

Volem una classe que emmagatzemi l'edat d'una persona, però només si és vàlida (0-150 anys).

ROL DEVELOPER: Escrivim el Codi

Perspectiva Developer

Pregunta que em faig: "Com faig que això funcioni i gestioni errors?"

Decisions que prenc:

  1. Creo una variable privada "edat"
  2. Creo un setter que valida abans d'assignar
  3. Si l'edat és invàlida, llanço IllegalArgumentException (és un paràmetre invàlid)
  4. Documento tot amb JavaDoc
/** * Classe que representa una persona amb validació d'edat */ public class Persona { // VARIABLE DE CLASSE: cada persona té la seva edat private int edat; /** * Estableix l'edat de la persona * * @param edat L'edat a assignar (ha de ser entre 0 i 150) * @throws IllegalArgumentException si edat menor que 0 o edat major que 150 */ public void setEdat(int edat) { // VALIDACIÓ 1: Detectar edats negatives // Per què IllegalArgumentException? Perquè "edat" és un PARÀMETRE invàlid if (edat < 0) { // "throw new" = crea i llança una excepció // Això ATURA el mètode i envia un error throw new IllegalArgumentException( "L'edat no pot ser negativa: " + edat ); } // VALIDACIÓ 2: Detectar edats no realistes if (edat > 150) { throw new IllegalArgumentException( "L'edat no és realista: " + edat ); } // Si arriba aquí, l'edat és vàlida // ASSIGNACIÓ: guarda el valor // "this.edat" = la variable d'AQUEST objecte Persona // "edat" (sense this) = el paràmetre que ens han passat this.edat = edat; } /** * Obté l'edat actual de la persona * * @return L'edat de la persona */ public int getEdat() { return this.edat; } }

Per què "this.edat = edat"?

this és una referència a l'objecte actual. És necessària per diferenciar:

  • this.edat = la variable de l'objecte (atribut de classe)
  • edat (sense this) = el paràmetre del mètode

Sense "this", Java no sap a quina variable et refereixes i el codi no funcionaria!

ROL TESTER: Escrivim els Tests

Perspectiva Tester

Pregunta que em faig: "Com puc verificar que això funciona? Què pot trencar-se?"

Casos de prova que dissenyò:

  1. Edat vàlida (exemple: 25) - hauria de funcionar
  2. Edat negativa (exemple: -5) - hauria de llançar excepció
  3. Edat massa gran (exemple: 200) - hauria de llançar excepció
  4. Casos límit: 0 i 150 - haurien de funcionar
import static org.junit.jupiter.api.Assertions.*; import org.junit.jupiter.api.Test; /** * Tests per verificar el comportament de la classe Persona */ public class PersonaTest { // TEST 1: Verificar que una edat vàlida s'assigna correctament /** * Test que verifica l'assignació d'una edat vàlida */ @Test public void testEdatValida() { // Arrange: Crear una persona Persona persona = new Persona(); // Act: Assignar una edat vàlida (25 està entre 0 i 150) persona.setEdat(25); // Assert: Verificar que s'ha guardat correctament assertEquals(25, persona.getEdat(), "L'edat hauria de ser 25"); } // TEST 2: Verificar que edat negativa llança excepció /** * Test que verifica que es llança excepció amb edat negativa */ @Test public void testEdatNegativa() { // Arrange Persona persona = new Persona(); // Act + Assert: Verificar que llança la excepció correcta // assertThrows espera: // 1r paràmetre: tipus d'excepció que esperem // 2n paràmetre: codi que hauria de llançar-la (lambda) Exception exception = assertThrows( IllegalArgumentException.class, () -> persona.setEdat(-5) ); // Extra: Verificar que el missatge d'error és correcte assertTrue( exception.getMessage().contains("negativa"), "El missatge hauria de mencionar 'negativa'" ); } // TEST 3: Verificar que edat massa gran llança excepció @Test public void testEdatMassaGran() { Persona persona = new Persona(); assertThrows( IllegalArgumentException.class, () -> persona.setEdat(200), "Edat 200 hauria de llançar excepció" ); } // TEST 4: Verificar casos límit (boundary values) @Test public void testCasosLimit() { Persona persona = new Persona(); // assertAll executa TOTS els asserts, encara que algun falli // Així veiem tots els problemes d'un cop assertAll( () -> { persona.setEdat(0); assertEquals(0, persona.getEdat(), "Edat 0 hauria de ser vàlida"); }, () -> { persona.setEdat(150); assertEquals(150, persona.getEdat(), "Edat 150 hauria de ser vàlida"); } ); } }

Veus la diferència?

Developer: "Faig que funcioni i gestioni errors"

Tester: "Verifico que funciona i que gestiona errors"

Resultat: Codi robust i ben testat

4. Assertions Avançades per Testers

Ara que entens el rol de tester, vegem TOTES les eines que tens disponibles.

assertAll()

Què fa: Agrupa múltiples verificacions i les executa TOTES

Per què l'uso: Vull veure TOTS els errors d'un cop, no només el primer

Quan l'uso: Quan vull verificar diverses coses relacionades

// Exemple: Verificar múltiples propietats assertAll( () -> assertEquals(5, suma), () -> assertTrue(suma > 0), () -> assertFalse(suma < 0) );

assertThrows()

Què fa: Verifica que el codi llança una excepció específica

Per què l'uso: Per comprovar que el codi detecta errors correctament

Quan l'uso: Per testejar tots els casos d'error

// Captura l'excepció per verificar-la Exception exc = assertThrows( IllegalArgumentException.class, () -> validador.validar(null) ); // Verificar el missatge assertTrue(exc.getMessage().contains("null"));

Missatges personalitzats

Què fa: Afegeix una explicació clara quan el test falla

Per què l'uso: Per entendre ràpidament què ha fallat sense haver de llegir tot el codi

Quan l'uso: SEMPRE! Facilita molt el debugging

// Sense missatge (poc informatiu) assertEquals(10, total); // Amb missatge (molt millor!) assertEquals(10, total, "La suma de 3+7 hauria de ser 10" );

assertTimeout()

Què fa: Verifica que el codi s'executa en un temps determinat

Per què l'uso: Per assegurar que no hi ha problemes de rendiment

Quan l'uso: Quan el rendiment és crític

import java.time.Duration; assertTimeout( Duration.ofSeconds(2), () -> processarDades() );

Diferència CLAU: Developer vs Tester

Situació Developer usa Tester usa
Llançar una excepció throw new IllegalArgumentException() -
Verificar que es llança - assertThrows(IllegalArgumentException.class, ...)
Gestionar un error try { ... } catch { ... } -
Verificar un valor - assertEquals(...)

5. Documentació amb JavaDoc

Tant developers com testers han de documentar el seu codi. JavaDoc és l'estàndard de Java.

Sintaxi completa de JavaDoc

/** * Descripció breu del que fa el mètode (una frase) * * Aquí pots afegir més detalls si cal: * - Comportament especial * - Casos límit * - Exemples d'ús * * @param nomParametre Descripció del paràmetre * @return Descripció del valor que retorna * @throws TipusExcepcio Descripció de QUAN es llança */

Tags de JavaDoc més importants

Tag Quan usar-lo Exemple
@param Per cada paràmetre del mètode @param edat L'edat de la persona (0-150)
@return Si el mètode retorna algo @return true si l'edat és vàlida, false en cas contrari
@throws Per cada tipus d'excepció que pot llançar @throws IllegalArgumentException si edat < 0
Developer

Documento el codi de producció

/** * Valida el format d'un email * * Comprova que l'email conté @ * i un punt després de l'@ * * @param email Email a validar * @return true si format vàlid * @throws IllegalArgumentException * si email és null */ public boolean validar(String email)
Tester

Documento els tests

/** * Verifica que validar() llança * IllegalArgumentException quan * l'email és null */ @Test public void testEmailNull() { assertThrows( IllegalArgumentException.class, () -> validador.validar(null) ); }

Com generar documentació HTML des de JavaDoc

Els IDEs poden generar documentació automàticament:

  • IntelliJ IDEA: Tools → Generate JavaDoc
  • Eclipse: Project → Generate Javadoc
  • Terminal: javadoc *.java

Això crea fitxers HTML que pots obrir al navegador!

6. Resum: Què has après

Concepte Developer Tester
Excepcions Llanço amb "throw new TipusExcepcio()" Verifico amb "assertThrows(TipusExcepcio.class, ...)"
IllegalArgumentException Uso quan un paràmetre té valor invàlid Espero quan testem validacions de paràmetres
this Diferencio variable de classe vs paràmetre No aplicable
assertAll No aplicable Agrupo múltiples verificacions relacionades
Missatges personalitzats Els poso a les excepcions Els poso als asserts per facilitar debugging
JavaDoc Documento codi: què fa, paràmetres, retorn, excepcions Documento tests: què verifiquen
